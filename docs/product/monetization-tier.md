# Monetization Tier

## 目标

在不影响早期验证的前提下，建立可测试、可转化、可扩展的收费模型。

当前仓库已经有适合收费分层的基础能力：

- 用户体系：`high-eq-front/client/src/contexts/AuthContext.tsx`
- 核心生成：`high-eq-front/client/src/pages/ReplyApp.tsx`
- 历史记录：`high-eq-front/client/src/pages/History.tsx`
- 收藏沉淀：`high-eq-front/client/src/pages/Favorites.tsx`

同时，首页已增加一版价格验证区：

- 数据定义：`high-eq-front/client/src/data/pricingPlans.ts`
- 展示入口：`high-eq-front/client/src/pages/Home.tsx`

## 收费结构

### 免费版

适用人群：

- 第一次体验的新用户
- 低频使用者
- 还没建立使用习惯的人

建议权益：

- 每日少量基础生成
- 标准语气与通用模板
- 基础历史与收藏

建议限制：

- 限每日生成次数
- 不开放高级模板
- 不开放个性化画像
- 不开放多轮改写

### Pro 版

适用人群：

- 高频沟通用户
- 对结果质量和一致性要求高的人
- 有明显职场或关系沟通痛点的人

建议权益：

- 更高或接近不限的额度
- 多轮优化与改写
- 自定义角色与个人画像
- 模板库全量访问
- 收藏搜索、标签、复用

### 点数包

适用人群：

- 不想订阅的低频用户
- 在特定场景下有强烈需求的用户

建议权益：

- 按次购买生成额度
- 可以和免费版并存
- 可以作为 Pro 之外的补充包

## 权益边界建议

| 功能 | 免费版 | Pro 版 | 点数包 |
|------|--------|--------|--------|
| 基础生成 | 有 | 有 | 有 |
| 高频模板场景 | 部分 | 全量 | 部分 |
| 多轮改写 | 无 | 有 | 有限 |
| 自定义角色 | 无 | 有 | 无 |
| 收藏搜索与标签 | 无 | 有 | 无 |
| 高级复杂场景 | 无 | 有 | 有限 |
| 额度上限 | 低 | 高 | 按购买量 |

## 推荐验证顺序

### 第一阶段：先验证升级意愿

不要先接支付，先验证用户是否对权益感兴趣。

建议方式：

- 首页价格区点击率
- 生成完成后展示 Pro 权益卡片
- 在触达限制时展示升级说明

关注指标：

- `view_pricing`
- `click_pro_cta`
- `reach_free_limit`
- `upgrade_intent`

### 第二阶段：再验证价格敏感度

当升级兴趣足够明显时，再测试价格。

建议测试：

- 月付低门槛
- 年付折扣
- 点数包小额尝试

### 第三阶段：最后接支付

后端建议新增：

- `subscription_plan`
- `user_subscription`
- `usage_quota`
- `payment_order`
- `payment_callback_log`

## 后端模块建议

### subscription

- 套餐定义
- 套餐权益
- 用户当前订阅状态

### usage

- 每日免费额度
- Pro 额度
- 点数包额度
- 扣减与回滚

### payment

- 创建订单
- 支付回调
- 状态同步
- 对账记录

## 产品规则建议

1. 免费版不要过早限制到无法体验价值。
2. Pro 版必须强调“更像你本人”和“更省时间”，而不是只有更多次数。
3. 点数包主要用于降低首次付费门槛，不应替代订阅。
4. 升级提示最好出现在用户已经感知价值之后，比如复制、收藏、多轮优化时。

## 验收标准

- 用户能明确理解三个方案分别适合谁
- 首页存在清晰的价值分层入口
- 后续接支付时不需要重新定义整套权益逻辑
